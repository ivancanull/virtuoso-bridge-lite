\begin{abstract}
Agent-driven analog and mixed-signal (AMS) workflows require a stable software boundary between Python code and Cadence Virtuoso that supports remote access, multimodal feedback, and repeated autonomous use. This paper presents Virtuoso-Bridge, an open-source bridge built on the standard SKILL IPC mechanism (\texttt{ipcBeginProcess} + \texttt{evalstring}). The architecture separates SKILL execution (VirtuosoClient) from transport (SSHClient) and simulation (SpectreSimulator), with all SSH traffic (port forwarding, shell commands, and file transfer) multiplexed over a single persistent connection. Three API levels (raw SKILL, script injection, Pythonic layout/schematic editors) and integrated Spectre simulation with PSF parsing provide end-to-end coverage from schematic entry to post-simulation analysis. Multi-profile support enables agent teams to operate on multiple design servers in parallel. Together with CLI lifecycle management, screenshot-based visual verification, and 30 runnable examples, Virtuoso-Bridge provides a ready-to-use infrastructure layer that allows LLM-based agents to autonomously drive Cadence Virtuoso workflows without manual GUI interaction.
\end{abstract}

\begin{IEEEkeywords}
Cadence Virtuoso, agent-native tooling, Python automation, remote access, SSH tunneling, analog IC design, EDA workflows.
\end{IEEEkeywords}

\section{Introduction}
%═══════════════════════════════════════════════════════════════════════
\IEEEPARstart{T}{he} core mechanism for programmatic access to Cadence Virtuoso is well established. The SKILL language provides \texttt{ipcBeginProcess}, which spawns a child process inside the Virtuoso runtime and connects it through stdin/stdout pipes. The child process receives SKILL expressions, and Virtuoso evaluates them via \texttt{evalstring}. Results flow back through the same pipe.

SkillBridge~\cite{skillbridge} and Virtuoso-Bridge both build on this same IPC pattern, in which a child process receives SKILL expressions and returns evaluated results through the Virtuoso IPC channel.

The key design consideration is therefore not the execution mechanism itself, since that mechanism already exists, but the system boundary built around it. SkillBridge answers this question with a transparent Pythonic RPC layer in which Python objects map to SKILL objects, function calls map to SKILL calls, and the user interacts as if writing SKILL in Python syntax. This model provides effective support for interactive local exploration.

By contrast, Virtuoso-Bridge adopts an agent-native execution boundary that packages bridge lifecycle, remote transport, editing operations, and artifact return into a single project-local system. The resulting tool emphasizes four practical properties: transparent remote access through SSH tunnels, multiple programming levels from raw SKILL to Python APIs, feedback channels for visual and numerical checking, and lifecycle support for repeated agent use.

This work was motivated by our experience building AMS-IO-Agent~\cite{amsioagent}, an LLM-based agent for I/O ring design that was validated in a CMOS tape-out. In that project, the lack of a stable, remote-capable, and agent-friendly bridge between the agent and Virtuoso emerged as a central infrastructure requirement. Virtuoso-Bridge is the distilled and generalized result of that effort. The goal of this paper is to report the design, scope, and practical usage model of the resulting tool rather than to introduce a new execution primitive.

\section{Design and Architecture}
%═══════════════════════════════════════════════════════════════════════

The central design choice is the separation of \emph{what to execute} from \emph{how to reach the execution environment}. This is realized through two independent modules.

\begin{figure*}[!t]
\centering
\includegraphics[width=\textwidth]{figures/arch.pdf}
\caption{System architecture of virtuoso-bridge. On the user machine, CLI and Python code interact with VirtuosoClient and SpectreSimulator. SSHClient maintains a persistent ControlMaster connection that multiplexes three channels: TCP port-forwarding (SKILL execution via the remote daemon), SSH shell commands (Spectre invocation), and rsync file transfer (netlists, results, SKILL scripts). In local mode, SSHClient is bypassed and VirtuosoClient connects directly to the daemon. The stacked instances illustrate multi-profile support, where each profile manages an independent connection to a separate design server. An optional jump host can be configured for environments requiring bastion access.}
\label{fig:architecture}
\end{figure*}

\textbf{VirtuosoClient} is a pure TCP client. It connects to \texttt{localhost:port}, sends SKILL code as a JSON payload, and receives results with binary framing using STX/NAK delimiters. It is agnostic to SSH, remote hosts, and file transfer. User Python code calls VirtuosoClient first; when a tunnel is attached, file upload, download, and \texttt{load\_il()} preparation delegate to SSHClient, while SKILL execution still goes only through the TCP daemon endpoint.

\textbf{SSHClient} is a tunnel and deployment manager. In remote mode it establishes an SSH port forward, uploads the daemon and setup files, and manages file transfer. In local mode it is absent: the user loads the local IL file in the Virtuoso CIW, and \texttt{VirtuosoClient.local()} connects directly to the local daemon. Internally, SSHClient delegates all SSH operations (command execution, file transfer, and port forwarding) to a single \textbf{SSHRunner} layer that uses OpenSSH ControlMaster multiplexing. This means all SSH traffic rides on one TCP connection regardless of how many operations are interleaved, which substantially reduces handshake overhead when jump hosts are involved.

This decoupling allows VirtuosoClient to be tested without SSH infrastructure, and changing the transport from SSH to VPN or direct LAN requires no client code changes. The system also supports \textbf{multiple connection profiles}: by suffixing environment variables (e.g.\ \texttt{VB\_REMOTE\_HOST\_gpu1}), users can manage connections to several remote machines simultaneously from a single project, each with independent tunnel state and lifecycle. This enables agent-team workflows in which multiple agents operate on different design servers in parallel, each through its own bridge instance.

The communication protocol is deliberately simple: one TCP connection per request, a JSON object containing the SKILL string and timeout, and binary-framed responses (\texttt{0x02}/STX for success, \texttt{0x15}/NAK for error, \texttt{0x1E}/RS as terminator). Because Virtuoso's \texttt{ipcBeginProcess} fires the data handler once per line, the daemon flattens multi-line SKILL expressions into a single line before writing to the IPC pipe, stripping SKILL comments (which use \texttt{;} line-comment syntax) while preserving semicolons inside string literals. The flattened expression is then evaluated via \texttt{evalstring}; results flow back through the same pipe. The \texttt{core/} directory contains the entire bridge mechanism in 3 files.

\section{API Levels}
%═══════════════════════════════════════════════════════════════════════

A single interaction granularity is insufficient for AMS workflows. Virtuoso-Bridge provides three levels, and an agent can switch among them as needed.

\subsection{Level 1: Raw SKILL Execution}
The method \texttt{client.execute\_skill()} sends an arbitrary SKILL expression to Virtuoso and returns the result, equivalent to typing in the CIW console.

\begin{lstlisting}
client.execute_skill('printf("Hello!")')
client.execute_skill("getVersion()")
client.execute_skill("getHostName()")
\end{lstlisting}

Batch execution via \texttt{execute\_operations} reduces round-trip overhead.

\subsection{Level 2: Script Injection}
The method \texttt{client.load\_il()} uploads a SKILL file to the remote host and loads it into the running Virtuoso session. This level is suitable for encapsulated operations packaged once in reusable helper IL files.

\begin{lstlisting}
client.load_il("list_library_cells.il")
client.load_il("screenshot.il")
\end{lstlisting}

\subsection{Level 3: Pythonic APIs}
Context-managed editors abstract common schematic and layout patterns.

Schematic example:

\begin{lstlisting}
with client.schematic.edit(lib, cell) as sch:
    sch.add_instance("analogLib", "res", (0.0, 0.0), name="R0")
    sch.add_wire([(0.0, 0.0), (1.2, 0.0)])
    sch.add_pin("OUT", "output", (1.2, 0.0))
\end{lstlisting}

Layout example:

\begin{lstlisting}
with client.layout.edit(lib, cell) as ly:
    ly.add_rect("M1", "drawing", (1.0, 0.0, 2.0, 0.5))
    ly.add_path("M2", "drawing", [(1.0, 0.25), (2.5, 0.25)], width=0.1)
    ly.add_via("M1_M2", (1.5, 0.25))
\end{lstlisting}

The editor translates these calls into SKILL and handles open/save/close automatically. This level is most suitable for AI agents because it reduces the amount of SKILL syntax they must generate directly.

\section{Feedback Channels}
%═══════════════════════════════════════════════════════════════════════

An agent operating in an EDA environment cannot rely solely on textual return values. Layout correctness, schematic topology, and waveform shapes are inherently visual. Virtuoso-Bridge provides several multimodal return paths.

\paragraph{Screenshot-based visual verification.} A SKILL helper script captures the current Virtuoso window, transfers it to the local machine via rsync, and returns the image to the agent. When combined with a vision-language model (VLM), the agent can inspect the result and decide whether a layout or schematic edit is correct before proceeding.

\paragraph{Simulation data.} Spectre results are parsed from PSF ASCII into Python data structures, supporting both real-valued (transient, DC) and complex-valued (AC) waveforms. The agent can compute derived metrics such as $f_\mathrm{3dB}$ from a frequency response and make optimization decisions programmatically.

\paragraph{File transfer and logging.} Arbitrary files can be uploaded to or downloaded from the remote host via rsync. All SSH and SKILL commands are logged with timestamps for full traceability and post-hoc analysis of agent behavior. These return paths collectively enable closed-loop iteration.

\section{Agent-Native Features}
%═══════════════════════════════════════════════════════════════════════

An agent-native system must support autonomous, repeated tool use without human setup at each round.

\paragraph{CLI-first lifecycle.} The bridge lifecycle is managed through \texttt{virtuoso-bridge init/start/status/restart/stop/license}. The \texttt{license} command queries \texttt{lmstat} on the remote host and reports all Cadence features with active users. All commands accept a \texttt{-{}-profile} flag for multi-machine scenarios.

\paragraph{Skill files.} The \texttt{skills/} directory provides progressive-disclosure documents: a concise \texttt{SKILL.md} loaded into the agent's context establishes the mental model, while detailed API references in \texttt{references/} subdirectories are loaded on demand.

\paragraph{Persistent tunnel.} The SSH tunnel is detached from the parent process and survives across script invocations. Tunnel state (PID, port, remote host) is persisted to a JSON file and reused by subsequent connections, eliminating repeated setup cost for high-frequency agent interaction.

\paragraph{Runnable examples.} Over 30 examples cover layout editing, schematic creation, CDL import (\texttt{spiceIn}), Spectre netlist export, ADE Assembler (Maestro) sweeps and spec checking, and Spectre simulation (transient, DC+AC, PSS+Pnoise). These serve as both documentation and few-shot references for agents.

\section{Simulation Integration}
%═══════════════════════════════════════════════════════════════════════

Spectre simulation is integrated as a first-class capability.

\begin{lstlisting}
sim = SpectreSimulator.from_env(work_dir="./out")
result = sim.run_simulation("tb.scs", {})
\end{lstlisting}

The system sources the Cadence environment on the remote host, uploads netlists and include files (including Verilog-A modules), executes Spectre via SSH shell commands, downloads PSF results, and parses them into Python data structures. Eight simulation modes are supported: basic Spectre, APS, and Spectre~X variants (CX, AX, MX, LX, VX), selectable via a single parameter. In remote mode, \texttt{SpectreSimulator.from\_env()} reuses the ControlMaster connection established by \texttt{virtuoso-bridge start}. A \texttt{SpectreSimulator.local()} factory method is also provided for local execution without SSH.

Both systems use the same SKILL IPC calls (\texttt{ipcBeginProcess}, \texttt{evalstring}, \texttt{ipcWriteProcess}). SkillBridge provides transparent Python-to-SKILL type mapping and IDE tab completion for interactive local exploration. Virtuoso-Bridge adds SSH remote access, multi-profile connections for agent-team parallelism, structured layout/schematic APIs, Spectre simulation integration, multimodal feedback, and agent-native lifecycle management. \section{Scope of Supported Workflows}
%═══════════════════════════════════════════════════════════════════════

The bridge supports remote operation (SSH with optional jump host) and local operation, with the same VirtuosoClient interface in both modes. It targets macOS, Windows, and Linux. In remote mode, SSH access must be pre-configured and a Virtuoso process must already be running; first-time use requires loading a setup script in the CIW.

\paragraph{Workflow coverage.} The repository includes runnable examples for raw SKILL execution, loading external \texttt{.il} files, layout editing, schematic editing, screenshot-based visual checking, CDL import via \texttt{spiceIn}, ADE Assembler (Maestro) control through \texttt{mae*} SKILL APIs, and Spectre simulation flows including transient, DC+AC, and PSS+Pnoise analyses. Most examples require only \texttt{analogLib} and run without a specific PDK.

\paragraph{Design boundaries.} Virtuoso-Bridge does not provide full Python-to-SKILL bidirectional type mapping. The fuller harness from which it is distilled was used in AMS-IO-Agent~\cite{amsioagent}, where an LLM-based agent completed I/O ring design tasks contributing to a silicon tape-out. The open-source release preserves the core interfaces but does not reproduce every environment-specific detail from the internal deployment.

\section{Conclusion}
%═══════════════════════════════════════════════════════════════════════

Virtuoso-Bridge provides a structured encapsulation of the standard SKILL IPC facility through an agent-native tool boundary. The architecture separates SKILL execution (VirtuosoClient) from transport (SSHClient), with all SSH traffic (port forwarding for SKILL execution, shell commands for Spectre invocation, and rsync for file transfer) multiplexed over a single persistent ControlMaster connection. Three levels of programmability (raw SKILL, script injection, Pythonic APIs) accommodate workflows ranging from low-level device parameter manipulation to high-level layout and schematic editing. Multimodal feedback through screenshots, parsed simulation data, and file transfer enables closed-loop agent iteration without human visual inspection.

Multi-profile support allows agent teams to operate on multiple design servers in parallel, each through an independent bridge instance with its own tunnel state and lifecycle. This is particularly valuable in scenarios where different agents in a team specialize in distinct subtasks, for example, one agent performing schematic design on one server while another runs Spectre simulations on a separate machine, enabling concurrent workflows that would otherwise require sequential access to a single Virtuoso session.

The bridge has been validated across macOS, Windows, and Linux, supporting both local and remote workflows. Its design prioritizes operational simplicity: a one-time CIW setup, CLI-driven lifecycle management, and automatic tunnel persistence ensure that agents can begin productive work with minimal configuration overhead.

Looking ahead, several directions merit further exploration. First, the current string-based type system trades off convenience for transparency; a lightweight type-mapping layer could reduce boilerplate in agent-generated SKILL while preserving inspectability. Second, extending the multi-profile model to support dynamic server allocation, where agents request design server capacity on demand rather than binding to pre-configured profiles, would better serve elastic agent-team topologies. Third, deeper integration with DRC/LVS checking tools would close additional verification loops that currently require manual intervention. Finally, providing a structured event or callback mechanism, where the bridge notifies the agent of asynchronous Virtuoso events such as simulation completion or DRC violations, would reduce polling overhead and enable more reactive agent architectures.

The system is open-source and publicly available.
